1. Abstract

Migration Reporting Tool is a web-based utility. Designed as a solution for visualizing, editing, and reporting migration data processes, the tool integrates user observations and data from various sensors to generate, in real-time, an interactive map depicting migrations conducted by different entities.

2. Introduction

Migration Reporting Tool (MIR) is a project proposed at Web Application Development course. In order to resolve this task, the REST paradigm will be used. The communication between the server and the clients will be resolved by creating an REST API that will handle the desired comunnication in our web application.The migrations will be shown on an interactive map and they will be handled in different ways by the users, regarding the role of the client.

3. Motivation

We opt for this project because it is an interisting problem in real life, regarding some situations: firstly, such a tool will be useful to see how Romanians go abroad because of the financial situation. Then, we taught about the migrations that took place because of the wars. Finally, we can see the impact of such an application when we want to track how birds or some species of animals migrates because of the weather/ climate changes.

4.2. External Data Sources

MIR will use external data sources such as:

  • Migration Data API - that will help us in managing the functionalities
  • Geolocation Services - that will help us in showing the migrations on a map
    • The interactive map will be implemented using Mapbox since it fits all criterias we are looking for.
  • Stardog API
    • This API is instrumental in managing core functionalities within MIR, such as updating, retrieving, and analyzing migration data. It serves as a crucial backbone for our data management strategy, ensuring that our repository remains up-to-date and accurate. Stardog has all the tools necessarry to upload a rdf schema, more than that, we could just create an endpoint to which our application could connect and make the necessarry API calls such: creating a migration, gathering all the migrations for a specific user and so on. All of these are possible thanks to a SPARQL endpoint implementation.
  • AWS s3 API
    • This API allows a user to upload a photo to the S3 bucket. The project doens't implement a secure way to communicate with the bucket, since the implementation shows the access key and the secret key. But, other than that, thanks to this we can make use of photos.
  • Location Data
    • Thanks to some databases we could figure out a way to download a file which contains all the countries with their cities, along with that we have longitute and latitute for every city on the Earth. We make use of this information when we create or update a migration, but not only that, also when we compute migrations for the interactive map.

4.3 Conforming with linked data principles

The solution implemented in MIR will use linked data principles, ensuring data interoperability and accessibility. For our project, we think at these possible solutions:

  • Usage of Uniform Resource Identifiers (URIs) for data identification
  • Linking data elements through RDF (Resource Description Framework)

We have proposed the following models for using RDFs:

    1. Ontology IRI
      2. We we're looking for something meaningful, clear and sugestive, and we've choosen: http://www.semanticweb.org/web-proj/MIR as for our IRI
    2. Class
      3. We have the following classes:
        1. Migration
        2. MigrationType
        3. MigrationLocation
        4. MigrationContext
      These classes model the way we store the RDF.
    3. Object Properties - Relationships
        1. hasType
          2. Is the property of a migration that ensures a Migration must have a MigrationType.
        2. hasLocation
          3. Is the property of a migration that ensures a Migration must have a MigrationLocation.
        3. hasContext
          4. Is the property of a migration that ensures a Migration must have a MigrationContext.
    4. Attributes
        1. Migration attributes
          2. Migration class has the following attributes: id, userId, migrationLocationId, migrationContextId, migrationTypeId; where id is a unique identifier, userId is the link between the user that created the migration and migration, migrationLocationId is the link between the migrationLocation and migration, migrationContextId is the link between the migrationContext and migration and migrationTypeId is the link between migrationLocation and migration.
        2. MigrationLocation attributes
          3. MigrationLocation has the following attributes: id, country, city, longitude, latitude; where id is the unique identifier, country states the country of the migration, city states the city of the migration, longitude and latitude being the fields populated when both country and city have been selected.
        3. MigrationType attributes
          4. MigrationType has the following attributes: id, category; where id is the unique identifier and the category is useful for labeling a migration.
        4. MigrationContext attributes
          5. MigrationContext attributes has the following attributes: id, working, thumbnailUrl, season, politicFactors, name, description, calamity. All of them are attributes that put label on our migration.

4.4 Functionalities in depth

This section will describe the functionalities mentioned in 4.1
    1. Login
      2. A guest can log in to become a user of the application in order to have access to some functionalities as: create a migration, delete a migration, edit a migration, see the list of migrations a user has created.
    2. Register
      3. A guest can register to become a user of the application in order to have access to some of the functionalities as: create a migration, delete a migration, edit a migration, see the list of migrations a user has created. A guest will be restrictioned from accessing every functionality on the website, but he will have access to: interaction with map, seeing one migration in detail.
    3. CRUD on migration
      4. This can only be performed by a user throughout our website. They will be able to create, read, update and delete a migration interacting with our interface.
    4. See a list of migrations
      5. This can only be performed by a user. This functionality will allow the user to see all the migrations he has created giving giving him the posibility of editing and deleting the migration.
    5. Interaction with inteligent map
      6. Either an user or a guest can have access to this functionality. They will be able to interact with this map to see what others have posted.

6. Conclusion

In conclusion, the Migration Reporting Tool (MIR) represents a comprehensive solution for visualizing, managing, and reporting migration data processes. Throughout the development of this web-based utility, we have carefully considered various aspects, ranging from internal data structures to external data sources and adhering to linked data principles.

The functionalities of MIR cover a broad spectrum, including user authentication, CRUD operations on migrations, and interaction with an intelligent map. We have designed internal data models, employed external data sources, and ensured adherence to linked data principles for improved data interoperability and accessibility.

The incorporation of ontologies within the Migration Reporting Tool (MIR) marks a pivotal aspect of our project, contributing significantly to the overall structure, understanding, and interoperability of migration data. Through careful consideration and application of linked data principles, our ontologies provide a standardized and semantically rich framework for representing various entities and relationships within the system.

Structured Data: The ontologies, identified by the chosen Ontology IRI (http://www.semanticweb.org/web-proj/MIR), ensure structured and organized data representation. Classes such as Migration, MigrationType, MigrationLocation, and MigrationContext delineate the distinct components of migration data, enhancing clarity and maintainability.

Interconnected Relationships: Object properties like hasType, hasLocation, and hasContext establish interconnected relationships, reflecting the dependencies and associations between different elements. This not only aids in comprehensive data modeling but also fosters better understanding of migration processes.

Attributes and Semantic Enrichment: The attributes associated with each class provide a granular level of detail, enriching the semantics of the data. This semantic richness not only enhances the user experience but also facilitates seamless integration with external data sources, contributing to a more holistic view of migration phenomena.

The collaboration process has been facilitated through GitHub, where we collaboratively created diagrams, developed the OpenAPI specification, implemented Scholarly HTML, and tracked project progress. The deployment of the application using Microsoft Azure, for backend, and Firebase, for frontend, adds a practical dimension to our efforts.
